Step 1

Meet Your File

Think of your file like a book. BitTorrent doesn't send the whole book at once. Instead, it tears the book into hundreds of pages (pieces), then computes a unique "fingerprint" for each page — so anyone can verify they got the right one.

BitTorrent uses a divide-and-verify approach. A file is split into fixed-size pieces. Each piece gets a SHA-1 hash. These hashes are stored in the .torrent metainfo file. When a peer downloads a piece, it hashes it and compares to the expected hash — ensuring data integrity even when downloading from untrusted peers.

Step 2

Choose Your Piece Size

How big should each "page" of the book be? Smaller pages mean more pages to track, but you can download missing pages more precisely. Bigger pages are fewer but chunkier — great for large files where overhead matters.

Piece size is a tradeoff between overhead and granularity. Smaller pieces generate more SHA-1 hashes (larger .torrent file, more DHT requests), but enable finer-grained download scheduling and faster piece completion. Larger pieces reduce overhead but mean you must download more before verifying integrity. Typical values range from 256 KiB (small files) to 4 MiB (large files > 1 GB).

Step 4

Computing Piece Hashes

Each piece gets a unique "fingerprint" — a hash. Change even a single byte and the fingerprint changes completely. This is how peers detect corrupt or tampered data. It's like a tamper-evident seal on each page of your book.

BitTorrent uses SHA-1 to hash each piece. SHA-1 produces a 160-bit (20-byte, 40-hex-char) digest via the SubtleCrypto Web API — the same API your browser uses for HTTPS. When a leecher downloads a piece, it hashes the received bytes and compares against the expected hash from the .torrent file. A mismatch means the piece is corrupt and must be re-downloaded. This is the avalanche effect: tiny input changes produce completely different outputs.

Step 5

Torrent Metadata Structure

All the information about your file — its name, size, pieces, and all the fingerprints — gets bundled into a tiny metadata file: the .torrent file. This is what you share with others, not the file itself! Anyone with this file can find peers and verify every piece they download.

The .torrent file contains a bencoded dictionary. The critical info dictionary contains the file name, length, piece length, and concatenated SHA-1 hashes. The info-hash — SHA-1 of the bencoded info dictionary — uniquely identifies the torrent and is used as the key in the DHT (Distributed Hash Table). Trackers, magnet links (magnet:?xt=urn:btih:<info-hash>), and peers all reference this hash.

Step 6

Network Distribution Simulation

Now imagine thousands of people sharing your file. Everyone downloads and uploads simultaneously. The more popular a file, the faster it spreads. Your pieces are verified against the fingerprints before being shared on — bad data never propagates.

BitTorrent uses rarest-first piece selection — peers prefer downloading pieces that fewer peers have, ensuring availability across the swarm. Pieces are verified via SHA-1 before being uploaded to others. The tit-for-tat algorithm encourages uploading: peers who upload more get better download speeds ("unchoked"). Seeders have 100% of pieces; leechers download while simultaneously uploading what they have.
