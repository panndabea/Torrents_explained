Step 1

Meet Your File

💡
Why this matters: Understanding file structure helps you see why BitTorrent can resume interrupted downloads and verify data piece by piece.

Think of your file like a book. BitTorrent doesn't send the whole book at once. Instead, it tears the book into hundreds of pages (pieces), then computes a unique "fingerprint" for each page — so anyone can verify they got the right one.

BitTorrent uses a divide-and-verify approach. A file is split into fixed-size pieces. Each piece gets a SHA-1 hash. These hashes are stored in the .torrent metainfo file. When a peer downloads a piece, it hashes it and compares to the expected hash — ensuring data integrity even when downloading from untrusted peers.

BitTorrent clients compute SHA-1 over the raw piece bytes using the SubtleCrypto Web API (crypto.subtle.digest('SHA-1', buffer)). The 160-bit (20-byte) digest is stored in the .torrent info dictionary under the key pieces as a concatenated binary string. The info-hash — SHA-1 of the entire bencoded info dict — acts as the torrent's globally unique identifier used in DHT (Distributed Hash Table) lookups and magnet links.

Step 2

Choose Your Piece Size

💡
Why this matters: Piece size is the key tradeoff that determines download speed, overhead, and torrent file size.

How big should each "page" of the book be? Smaller pages mean more pages to track, but you can download missing pages more precisely. Bigger pages are fewer but chunkier — great for large files where overhead matters.

Piece size is a tradeoff between overhead and granularity. Smaller pieces generate more SHA-1 hashes (larger .torrent file, more DHT requests), but enable finer-grained download scheduling and faster piece completion. Larger pieces reduce overhead but mean you must download more before verifying integrity. Typical values range from 256 KiB (small files) to 4 MiB (large files > 1 GB).

The optimal piece size minimizes torrent_file_size = num_pieces × 20 bytes while keeping verification latency acceptable. With 16 KiB pieces a 1 GiB file requires 65,536 SHA-1 hashes (1.28 MiB of hash data), whereas 4 MiB pieces reduce that to 256 hashes (5 KiB). Most modern clients use power-of-two sizes between 512 KiB and 4 MiB. BitTorrent v2 switches to SHA-256 and a Merkle tree structure allowing per-block (16 KiB) verification without increasing torrent file size.

Step 3

Splitting Into Pieces

💡
Why this matters: Splitting allows many peers to exchange different parts in parallel, making downloads faster as more people join.

Watch your file get sliced into equal-sized pieces. Each colored block is one piece. Click any piece to see the actual bytes inside — this is the raw binary data that BitTorrent will distribute across the network!

The file's bytes are sliced into contiguous, non-overlapping segments. Each piece has a start and end byte offset. The last piece may be smaller than the piece length. Pieces are identified by a zero-based index. Click any piece to inspect its byte range and a hex dump of its first 32 bytes.

Piece boundaries are strictly byte-aligned and non-overlapping: piece i spans bytes [i × piece_length, min((i+1) × piece_length, file_size)). The last piece is often shorter. Clients use a bitmask (bitfield) to track which pieces they have; this bitfield is exchanged in the BitTorrent handshake. Piece availability is distributed non-uniformly: the rarest piece is prioritized to maintain swarm health (rarest-first algorithm).

Step 4

Computing Piece Hashes

💡
Why this matters: Hashes let clients instantly detect corrupted or tampered data — bad pieces are rejected before they spread.

Each piece gets a unique "fingerprint" — a hash. Change even a single byte and the fingerprint changes completely. This is how peers detect corrupt or tampered data. It's like a tamper-evident seal on each page of your book.

BitTorrent uses SHA-1 to hash each piece. SHA-1 produces a 160-bit (20-byte, 40-hex-char) digest via the SubtleCrypto Web API — the same API your browser uses for HTTPS. When a leecher downloads a piece, it hashes the received bytes and compares against the expected hash from the .torrent file. A mismatch means the piece is corrupt and must be re-downloaded. This is the avalanche effect: tiny input changes produce completely different outputs.

SHA-1 processes data in 512-bit (64-byte) blocks through 80 rounds of mixing operations producing a 160-bit digest. The avalanche effect guarantees that flipping a single input bit changes ~50% of output bits — making it infeasible to forge a piece that passes verification. While SHA-1 has known collision vulnerabilities in cryptographic contexts, its use in BitTorrent (integrity checking, not authentication) remains practically sound. BitTorrent v2 upgrades to SHA-256 for stronger guarantees.

Step 5

Torrent Metadata Structure

💡
Why this matters: The .torrent file describes how to rebuild the original file without containing it — it's a recipe, not the food.

All the information about your file — its name, size, pieces, and all the fingerprints — gets bundled into a tiny metadata file: the .torrent file. This is what you share with others, not the file itself! Anyone with this file can find peers and verify every piece they download.

The .torrent file contains a bencoded dictionary. The critical info dictionary contains the file name, length, piece length, and concatenated SHA-1 hashes. The info-hash — SHA-1 of the bencoded info dictionary — uniquely identifies the torrent and is used as the key in the DHT (Distributed Hash Table). Trackers, magnet links (magnet:?xt=urn:btih:<info-hash>), and peers all reference this hash.

The info-hash is computed as SHA-1 of the bencoded info dictionary — a deterministic encoding where dictionary keys are sorted lexicographically. Even adding a single space to the file name produces a completely different info-hash, making it impossible to accidentally collide. Magnet links encode the info-hash as magnet:?xt=urn:btih:<40-hex-or-32-base32>. The DHT stores peers under this key. Trackers use it to group peers. The info-hash is the torrent's global identity.

Step 6

Network Distribution Simulation

💡
Why this matters: The swarm model means every downloader also becomes an uploader — the more popular a file, the faster it spreads.

Now imagine thousands of people sharing your file. Everyone downloads and uploads simultaneously. The more popular a file, the faster it spreads. Your pieces are verified against the fingerprints before being shared on — bad data never propagates.

BitTorrent uses rarest-first piece selection — peers prefer downloading pieces that fewer peers have, ensuring availability across the swarm. Pieces are verified via SHA-1 before being uploaded to others. The tit-for-tat algorithm encourages uploading: peers who upload more get better download speeds ("unchoked"). Seeders have 100% of pieces; leechers download while simultaneously uploading what they have.

The tit-for-tat algorithm works in 10-second rounds: each peer "unchokes" (allows uploads to) its top 3 uploaders plus one random "optimistic unchoke" to discover new fast peers. Choked peers can only send interested messages, not requests. The end-game mode triggers when a client has requested all remaining pieces: it sends duplicate requests to all peers for the last few pieces to minimize tail latency. DHT (BEP 5) allows trackerless peer discovery using a distributed key-value store.
